# Reproducible Builds: Technical Details

This page describes the Gradle tasks, evidence formats, and implementation
rules behind reproducible release-build verification.

## Task Model

`verifyReleaseBuild` is the main Java payload gate. It depends on:

- `verifyReleaseEnvironment`
- `verifyGradleWrapperSecurity`
- `verifyReleaseBuilderImage`
- `verifyGithubActionsSecurity`
- `verifyDependencySignaturePolicy`
- `verifyReleaseEvidenceBundle`
- every project `build` task
- every project `verifyReproducibleArchives` task

`verifyInstallerEvidenceBundle` is the main installer evidence gate. It depends
on `generateInstallerEvidenceBundle`, which depends on installer manifest and
installer structure report generation.

The comparison tasks are read-only:

- `compareReleaseEvidenceBundles`
- `compareInstallerEvidenceBundles`

Both accept ZIP bundles or extracted evidence directories.

## Java Release Evidence

`release-manifest.tsv` is the canonical Java payload comparison file. It
contains SHA-256, byte size, and canonical repo-relative path.

`release-evidence.zip` contains:

- `release-manifest.tsv`
- `SHA256SUMS`
- `build-info.json`
- `checksums/jars.sha256`

`build-info.json` is diagnostic metadata. It records Gradle and Java runtime
details, OS details, selected environment variables, release-builder Dockerfile
identity when present, and installer-relevant tool versions when available.
It explains differences but is not the canonical comparison file.

`compareReleaseEvidenceBundles` compares only the manifest entries. A mismatch
reports missing, extra, and changed paths with left and right hashes and sizes.

## Installer Evidence

`installer-manifest.tsv` is the canonical installer artifact comparison file.
It records SHA-256, byte size, and canonical repo-relative path for installer
files produced by `:desktop:generateInstallers`.

Expected formats are OS-specific:

- Linux: `.deb` and `.rpm`
- macOS: `.dmg`
- Windows: `.exe`

`installer-evidence.zip` contains:

- `installer-manifest.tsv`
- `INSTALLER-SHA256SUMS`
- `installer-build-info.json`
- `installer-structure-summary.txt`
- `installer-structure-report.tsv`
- referenced `installer-structure/*.txt` reports

`installer-structure-report.tsv` records whether each per-artifact structure
report was generated, failed, or skipped. Structure reports are diagnostic.
They are used to explain installer differences, while
`installer-manifest.tsv` remains the canonical comparison input.

`compareInstallerEvidenceBundles` compares installer manifests. When a changed
artifact has structure-report rows, the failure output points at the matching
structure reports to speed up investigation.

## Archive Determinism

Gradle archive outputs must use deterministic metadata:

- preserve file timestamps disabled
- reproducible file order enabled

The same rule applies in the `bitcoind` submodule so embedded bitcoind jars do
not differ only by ZIP metadata.

## Linux Release-Builder Image

The Linux release-builder image is defined in
`docker/release-builder/linux/Dockerfile`. It pins:

- `linux/amd64` Azul Zulu OpenJDK 21.0.6 base image by digest
- Ubuntu apt snapshot timestamp
- `SOURCE_DATE_EPOCH=0`
- `TZ=UTC`
- `LANG=C.UTF-8`
- `LC_ALL=C.UTF-8`
- `HOME=/tmp/bisq-home`
- `GRADLE_USER_HOME=/tmp/bisq-home/.gradle`

It installs Linux package and archive tools used for release and installer
evidence: `binutils`, `coreutils`, `cpio`, `dpkg`, `dpkg-dev`, `fakeroot`,
`file`, `git`, `gnupg`, `gzip`, `rpm`, `tar`, `unzip`, `xz-utils`, and `zip`.

`verifyReleaseBuilderImage` statically checks that the Dockerfile keeps these
release-sensitive choices pinned.

## Linux RPM Normalization

Linux RPM installers are generated by `jpackage` and then repacked
deterministically. The repack step extracts the jpackage RPM payload and
rebuilds it with `rpmbuild` using fixed RPM macros:

- source date epoch as build time
- clamped payload mtimes
- fixed build host
- deterministic gzip payload
- disabled generated build-id symlink entries
- disabled post-install buildroot processing

The generated spec preserves Bisq package metadata and the desktop menu
install/uninstall script behavior. This keeps the installable payload aligned
with jpackage while removing nondeterministic RPM header and payload metadata.

## Dependency Verification Maintenance

Refresh dependency verification metadata separately from ordinary code changes:

```bash
./gradlew refreshDependencyVerificationKeyring
./gradlew resolveAndVerifyDependencies --write-verification-metadata pgp,sha256
./gradlew verifyDependencySignaturePolicy
./gradlew dependencySignatureReport
```

New checksum-only artifacts must be reviewed before they are added to
`gradle/dependency-checksum-fallback-allowlist.tsv`. Each allowlist entry must
include the exact dependency, artifact file name, and review rationale. Keep
entries sorted by dependency and artifact name.

## Investigation Order

When evidence differs:

1. Compare the manifest rows first.
2. Confirm both builds used the same commit, submodule commits, Gradle version,
   Java version, timezone, locale, and OS/tool versions.
3. For Java payload mismatches, inspect the changed jar or runtime-library
   path named by `compareReleaseEvidenceBundles`.
4. For installer mismatches, inspect `installer-build-info.json`,
   `installer-structure-summary.txt`, `installer-structure-report.tsv`, and
   the per-artifact report under `installer-structure/`.
5. Change build logic only after the differing field has been identified.
